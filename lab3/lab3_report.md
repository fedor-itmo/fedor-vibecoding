## 1. Описание деплоя

### Какой способ деплоя выбран

Для развёртывания бота подготовлены **два варианта**:

1. **Railway** (облачный хостинг) — основной способ для постоянной работы
2. **Docker** — для локального запуска и переносимости

### Почему эти способы

- **Railway** — бесплатный тариф, деплой в один клик через GitHub, автоматический перезапуск при сбоях, поддержка переменных окружения через веб-интерфейс
- **Docker** — контейнеризация гарантирует одинаковую работу на любой машине, упрощает развёртывание на VPS, `docker-compose up -d` — и бот работает

## 2. Подготовка к деплою

### Файлы для деплоя

В проект добавлены следующие файлы:

**Procfile** (для Railway):
```
worker: python bot.py
```

**Dockerfile** (для Docker):
```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "bot.py"]
```

**docker-compose.yml**:
```yaml
version: '3.8'

services:
  bot:
    build: .
    env_file:
      - .env
    restart: unless-stopped
```

**Логирование** (уже в bot.py):
```python
import logging

logging.basicConfig(
    format="%(asctime)s - %(name)s - %(levelname)s - %(message)s",
    level=logging.INFO,
)
logger = logging.getLogger(__name__)
```

- [x] `.env` в `.gitignore` — токен не попадёт в git
- [x] `requirements.txt` содержит все зависимости с версиями

## 3. Процесс деплоя

### Вариант 1: Railway (основной)

1. **Загрузка на GitHub**
   ```bash
   git add .
   git commit -m "Prepare for deploy"
   git push origin main
   ```

2. **Настройка Railway**
   - Зарегистрировался на [railway.app](https://railway.app) через GitHub
   - Создал новый проект → «Deploy from GitHub repo»
   - Подключил репозиторий `fedor-vibecoding`

3. **Переменные окружения**
   - В настройках проекта на Railway добавил переменную `BOT_TOKEN`
   - Вставил токен бота из BotFather

4. **Деплой**
   - Railway автоматически определил Python-проект
   - Установил зависимости из `requirements.txt`
   - Запустил бота

### Вариант 2: Docker (локальный / VPS)

1. **Сборка и запуск**
   ```bash
   docker-compose up -d --build
   ```

2. **Проверка логов**
   ```bash
   docker-compose logs -f bot
   ```

3. **Остановка**
   ```bash
   docker-compose down
   ```

### С какими проблемами столкнулся

| Проблема | Решение |
|---|---|
| Railway не находил точку входа | Добавил `Procfile` с указанием `worker: python bot.py` |
| Бот использовал polling, а Railway ожидал HTTP-сервер | Тип сервиса изменён на `worker` (не `web`) — polling работает без HTTP |
| Файл `favorites.db` пропадал после перезапуска на Railway | Это ожидаемое поведение для бесплатного тарифа; для продакшена нужен внешний PostgreSQL |
| Docker: образ занимал много места с полным Python | Использован `python:3.11-slim` вместо полного образа — размер уменьшен в 3 раза |

## 7. Улучшения на основе фидбека

На основе собранных отзывов были реализованы **3 улучшения**:

### Улучшение 1: Сравнение погоды в двух городах

**Проблема:** Никита и Артем хотели сравнивать погоду, чтобы выбирать куда поехать.

**Решение:** Добавлена команда `/compare Город1, Город2` и кнопка «⚖️ Сравнить города» в меню.

Бот показывает погоду обоих городов рядом и подводит итог: где теплее, суше и тише.

![IMG_0224](https://github.com/user-attachments/assets/8ce03aea-56ca-4456-9cba-ed08c25bfd0e)

### Улучшение 2: UV-индекс и время рассвета/заката

**Проблема:** Соня хотела видеть UV-индекс и время рассвета/заката.

**Решение:** В текущую погоду добавлены:
- UV-индекс с интерпретацией (низкий / средний / высокий + рекомендация по крему SPF)
- Время восхода и заката

![IMG_0225](https://github.com/user-attachments/assets/d5e07640-029a-48a6-86ad-59a3524dafca)

### Улучшение 3: Советы по одежде в прогнозе

**Проблема:** Дмитрий заметил, что в прогнозе на 3 дня нет советов по одежде, хотя в текущей погоде они есть.

**Решение:** В каждый день прогноза добавлена краткая рекомендация по одежде (основная, без подробностей об осадках и ветре — чтобы не перегружать прогноз).
